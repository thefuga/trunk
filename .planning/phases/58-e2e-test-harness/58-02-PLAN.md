**e2e/specs/staging.e2e.js** (E2E-03):

```javascript
import { createLinearRepo, cleanupRepo } from '../helpers/fixture.js';
import { openRepo, waitForCommitGraph } from '../helpers/app.js';
import { writeFileSync } from 'fs';
import { join } from 'path';
```

Test suite `describe('Staging Workflow', () => { ... })`:

- `let repoDir`
- `before(async () => { ... })`: Call `repoDir = createLinearRepo(1)` (one commit). Call `await openRepo(repoDir)`. Call `await waitForCommitGraph()`.
- `after(async () => { ... })`: Call `cleanupRepo(repoDir)`.

Test cases:
1. `it('should show unstaged file after modifying repo', async () => { ... })`:
   - Write a new file into the fixture: `writeFileSync(join(repoDir, 'new-file.txt'), 'hello e2e')`
   - Wait for the app to detect the file change. The app has a filesystem watcher, so wait: `await browser.waitUntil(async () => { const files = await $$('[data-testid="staging-file"]'); return files.length > 0; }, { timeout: 15000, timeoutMsg: 'Expected unstaged file to appear' })`
   - Verify: `const files = await $$('[data-testid="staging-file"]'); expect(files.length).toBeGreaterThanOrEqual(1)`

2. `it('should stage a file by clicking its action button', async () => { ... })`:
   - The FileRow has an action button (the "+" button for staging). Since we only have `data-testid="staging-file"` on the row, we need to click the action button within it. Find the staging-file within the unstaged section, then find the button inside it:
   - `const unstagedSection = await $('[data-testid="staging-unstaged-section"]')`
   - `const fileRow = await unstagedSection.$('[data-testid="staging-file"]')`
   - Find the action button inside the file row. The FileRow component has a button with aria-label based on `actionLabel` prop. For unstaged files the actionLabel is "+" so look for the button: `const actionBtn = await fileRow.$('button')`
   - `await actionBtn.click()`
   - Wait for the file to move to staged section: `await browser.waitUntil(async () => { const stagedSection = await $('[data-testid="staging-staged-section"]'); const stagedFiles = await stagedSection.$$('[data-testid="staging-file"]'); return stagedFiles.length > 0; }, { timeout: 10000, timeoutMsg: 'Expected file to appear in staged section' })`

3. `it('should create a commit with subject', async () => { ... })`:

   - Wait for commit to appear in history: `await browser.waitUntil(async () => { const summaries = await $$('[data-testid="commit-row-summary"]'); const texts = await Promise.all(summaries.map(s => s.getText())); return texts.some(t => t.includes('E2E test commit')); }, { timeout: 15000, timeoutMsg: 'Expected new commit to appear in graph' })`

Test suite `describe('Branch Operations', () => { ... })`:

2. `it('should checkout a branch on double-click', async () => { ... })`:

   - `const branchRows = await $$('[data-testid="branch-row"]')`
   - `let targetRow; for (const row of branchRows) { const text = await row.getText(); if (text.includes('feature-a')) { targetRow = row; break; } }`
   - Double-click: `await targetRow.doubleClick()`
   - Wait for checkout to complete (the branch row should become the head -- shown in bold/accent color). Wait for a toast or for the sidebar to refresh:
   - `await browser.waitUntil(async () => { const rows = await $$('[data-testid="branch-row"]'); for (const r of rows) { const text = await r.getText(); if (text.includes('feature-a')) { const style = await r.$('div[role="button"]').getAttribute('style'); return style && style.includes('font-weight: 600') || style.includes('600'); } } return false; }, { timeout: 10000, timeoutMsg: 'Expected feature-a to become head branch' })`

3. `it('should create a new branch', async () => { ... })`:
   - Click the "+" create button in the local section: `const createBtn = await $('[data-testid="branch-section-create-btn"]'); await createBtn.click()`
   - Wait for input to appear: `const input = await $('[data-testid="branch-create-input"]'); await input.waitForExist({ timeout: 5000 })`
   - Type branch name and submit: `await input.setValue('e2e-test-branch'); await browser.keys('Enter')`
   - Wait for the new branch to appear in sidebar: `await browser.waitUntil(async () => { const rows = await $$('[data-testid="branch-row"]'); const texts = await Promise.all(rows.map(r => r.getText())); return texts.some(t => t.includes('e2e-test-branch')); }, { timeout: 10000, timeoutMsg: 'Expected new branch to appear' })`

4. `it('should delete a branch via IPC', async () => { ... })`:
   - Branch delete uses native context menu + native confirm dialog, both inaccessible to WebDriver. Use IPC directly:
   - `await browser.execute(async (path) => { await window.__TAURI_INTERNALS__.invoke('delete_branch', { path, branchName: 'feature-b' }); }, repoDir)`
   - Wait for sidebar to refresh. The sidebar auto-refreshes on events, but may need a manual trigger. Wait for feature-b to disappear:
   - `await browser.waitUntil(async () => { const rows = await $$('[data-testid="branch-row"]'); const texts = await Promise.all(rows.map(r => r.getText())); return !texts.some(t => t.includes('feature-b')); }, { timeout: 10000, timeoutMsg: 'Expected feature-b to be removed from sidebar' })`
   - Note: If the sidebar doesn't auto-refresh after direct IPC, force a refresh by calling `await browser.execute(async () => { window.dispatchEvent(new Event('focus')); })` before the waitUntil, since the app may listen for focus events to refresh.

All specs use WDIO's `waitForExist()` and `waitUntil()` -- never `browser.pause()` per anti-patterns guidance.

**Create `.github/workflows/e2e.yml`** per D-06, D-07, D-08:

```yaml
name: E2E

on:
  push:
    branches: [main]
  pull_request:

concurrency:
  group: e2e-${{ github.ref }}
  cancel-in-progress: true

permissions:
  contents: read

jobs:
  e2e:
    name: E2E Tests
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6

      - name: Install system dependencies
        run: |
          sudo apt-get update
          sudo apt-get install -y \
            libwebkit2gtk-4.1-dev \
            build-essential \
            curl \
            wget \
            file \
            libxdo-dev \
            libssl-dev \
            libayatana-appindicator3-dev \
            librsvg2-dev \
            webkit2gtk-driver \
            xvfb

      - uses: dtolnay/rust-toolchain@stable
      - uses: Swatinem/rust-cache@v2
        with:
          workspaces: "src-tauri -> target"
          save-if: ${{ github.ref == 'refs/heads/main' }}

      - uses: oven-sh/setup-bun@v2

      - name: Install frontend dependencies
        run: bun install --frozen-lockfile

      - name: Install E2E dependencies
        run: cd e2e && bun install --frozen-lockfile

      - name: Install tauri-driver
        run: cargo install tauri-driver --locked

      - name: Build debug binary
        run: bun run tauri build -- --debug --no-bundle

      - name: Run E2E tests
        env:
          E2E_SKIP_BUILD: "1"
        run: xvfb-run bun --cwd e2e run test
```

Key points per locked decisions:
- Separate workflow file per D-06 (not in ci.yml)
- Uses Xvfb per D-07 (`xvfb-run` wraps the test command)
- Triggers on push to main and PRs per D-08
- Runs in parallel with ci.yml (no `needs:` dependency on other workflows)
- `E2E_SKIP_BUILD: "1"` because the binary is built in a separate step (avoids double-build)
- Same system deps as ci.yml PLUS `webkit2gtk-driver` and `xvfb`
- Same action versions as ci.yml (checkout@v6, rust-toolchain@stable, rust-cache@v2, setup-bun@v2)
- Same Rust cache config as ci.yml
- Concurrency config follows benchmarks.yml pattern (cancel-in-progress)
