<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user → destructive action | A delete is irreversible; the UI must gate it behind a confirmation that cannot be bypassed |
| frontend → IPC | Panel writes (add_commit_comment / edit_comment / delete_comment) carry the comment id + text the panel holds |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-69-15 | Tampering / Repudiation | accidental or bypassable comment deletion | mitigate | delete is gated by the native plugin-dialog `ask` confirm (D-05); cancel writes nothing; the confirm is in the only delete path; a vitest test asserts cancel → no delete_comment, confirm → delete_comment by id |
| T-69-16 | Tampering | edit/delete hitting the wrong comment after a multi-tab change | mitigate | the panel targets by stable `id` (D-03) carried in the DTO, never by row position; the session-changed listener reloads after external mutations |
| T-69-17 | Denial of Service | jump on an unresolvable anchor navigating nowhere or throwing | mitigate | orphan rows (resolve result false) disable jump entirely and render a read-only badge (D-08); jumpTo returns early on a null/unresolvable anchor — never navigates, never errors |
| T-69-18 | Information Disclosure | inline-edit emit storms leaking state churn / perf degradation | accept | inline editing is local component state until Save; only the Save-time edit_comment emits session-changed (Plan 02) — no per-keystroke writes |
</threat_model>
