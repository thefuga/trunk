<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Phase 76 host → MessageEditor.open() | Host passes a "default message" string (from `.git/MERGE_MSG`, a constructed merge-branch string, or a constructed revert message). The string is content destined for a commit message, not arbitrary untrusted input — but the component still treats it as opaque text. |
| User keyboard → textarea | User can type any text into the textarea. The resolved value flows back to Phase 76, where it becomes a commit message via the Rust helper's temp file (Plan 02). |
| Backdrop click → cancel | Mouse interaction. The `e.target === e.currentTarget` guard limits cancel-via-backdrop to the outer wrapper, not inner clicks. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-75-01 | Tampering | textarea binding of default value | mitigate | Svelte's `bind:value` writes to `.value`, NOT `innerHTML`. Default message containing HTML/script-looking text is rendered as plain text — no XSS surface. Verified implicitly by the "pre-fills textarea with the default" test asserting on `.value` byte-equality including special characters. |
| T-75-02 | Tampering | backdrop click cancel | accept | Backdrop click resolving `null` is intentional UX (matches CLI behavior of killing `$EDITOR`). User can dismiss accidentally; no half-state is reachable because the modal renders only while `open()` is pending and clears `resolveFn` after resolution. This is intentional behavior, not an attack surface. |
| T-75-03 | Information Disclosure | message content in DOM | accept | Default messages and edited content live in the rendered DOM while the modal is open. This is unavoidable for an editor UI; no PII beyond what the user is about to commit publicly anyway. Caller's responsibility to avoid prefilling secrets into the editor (no upstream code path does this — the three Phase 76 callers all pre-fill from git-derived content). |
| T-75-04 | Repudiation | empty Save → null | mitigate | The contract "empty/whitespace → null" is enforced by `trim().length === 0` in `handleSubmit`. Covered by two tests (empty and whitespace-only). Phase 76 consumers can rely on this — they treat `null` as abort, not as "user saved empty string". |
| T-75-05 | DoS | unbounded textarea size | accept | Browser-level textarea has no explicit max length. A user pasting a huge string would consume memory but cannot affect other processes. Practical maximum is bounded by the commit message use case. No mitigation needed for v0.14. |

No `high`-severity threats in this plan. T-75-SC (supply chain) not applicable — no new npm packages installed.
</threat_model>
