### 🚧 v0.9 Multi-tab & Tree View (In Progress)

**Milestone Goal:** Enable working with multiple repositories via tabs and browsing files as directory trees instead of flat lists.

- [ ] **Phase 44: Backend State Scoping** — Make RunningOp per-repo so tabs can run concurrent remote operations

## Phase Details

### Phase 44: Backend State Scoping
**Goal**: Each open repository can run remote operations (fetch, push, pull) independently without blocking other repositories
**Depends on**: Nothing (first phase of v0.9)
**Requirements**: BACK-01
**Success Criteria** (what must be TRUE):
  1. User can push in one tab while fetching in another tab simultaneously
  2. Cancelling a remote operation in one tab does not affect operations in other tabs
  3. Closing a tab cleans up its filesystem watcher without affecting other tabs
**Plans**: 1 plan
Plans:
- [ ] 44-01-PLAN.md — Per-repo RunningOp, cancel scoping, force_close_repo command

### Phase 45: Frontend Tab Architecture
**Goal**: Users can open multiple repositories as independent tabs within a single window, each with fully isolated state
**Depends on**: Phase 44
**Requirements**: TAB-01, TAB-02, TAB-03, TAB-04, TAB-05, TAB-06, TAB-07
**Success Criteria** (what must be TRUE):
  1. User can open multiple repositories in separate tabs and switch between them with each tab preserving its own graph, staging, diff, and operation state
  2. User can create a new tab (Cmd+T) that shows the splash/project picker, close a tab (Cmd+W / X button), and switch tabs (Cmd+1-9 / Ctrl+Tab / Ctrl+Shift+Tab)
  3. Open tabs and the active tab persist across app relaunch — reopening the app restores the same tabs
  4. Background tabs with uncommitted changes display a dirty indicator (dot badge) on the tab
  5. Undo/redo, remote state, and all per-repo state are scoped to each tab — actions in one tab never affect another
**Plans**: TBD
**UI hint**: yes

### Phase 46: Tree View Data Layer
**Goal**: A tested, pure-logic utility transforms flat file paths into a compressed directory tree structure
**Depends on**: Nothing (independent of tab work)
**Requirements**: TREE-07
**Success Criteria** (what must be TRUE):
  1. buildTree produces a correct nested tree from flat file paths with directories sorted before files
  2. Single-child directory chains are compressed into combined paths (e.g. src/lib/ instead of src > lib)
  3. Unit tests cover edge cases: empty input, deeply nested paths, single-file repos, unicode filenames, paths with only directories
**Plans**: TBD

### Phase 47: Tree View UI Integration
**Goal**: Users can browse files as a directory tree (instead of flat list) in all file list contexts
**Depends on**: Phase 46
**Requirements**: TREE-01, TREE-02, TREE-03, TREE-04, TREE-05, TREE-06
**Success Criteria** (what must be TRUE):
  1. User can toggle between flat file list and directory tree view in the staging panel, commit diffs, and merge editor
  2. Directory nodes expand and collapse with chevron indicators via click
  3. Expand/collapse state survives status refreshes — staging a file does not collapse the tree
  4. User can navigate the tree with arrow keys (up/down to move, left/right to collapse/expand parent/child)
  5. View mode preference (flat vs tree) is persisted and restored across sessions
**Plans**: TBD
**UI hint**: yes

### Phase 48: Polish & Differentiators
**Goal**: Competitive-parity tab interactions and tree view power features that elevate UX beyond basic functionality
**Depends on**: Phase 45, Phase 47
**Requirements**: TAB-08, TAB-09, TAB-10, TREE-08, TREE-09, TREE-10
**Success Criteria** (what must be TRUE):
  1. User can right-click a tab to access Close Others, Close All, and Copy Path actions
  2. User can middle-click a tab to close it
  3. Opening a repository that is already open in a tab switches to the existing tab instead of creating a duplicate
  4. User can stage or unstage an entire directory by clicking an action on the directory node
  5. Directory nodes display file count badges and Expand All / Collapse All buttons are available in the file list header
**Plans**: TBD
**UI hint**: yes
